# Task 가이드

## 목적

Task는 **"어떤 코드를 작성하는가"** 를 정의하는 문서다.  
Plan의 설계를 Phase 단위로 분해하여, 전체를 한 번에 구현하지 않고 **단계적으로 쌓아 올리는 방식**으로 구현한다.

Task가 완성되면 개발자는 어떤 Phase부터 시작할지, 각 Phase에서 어떤 파일을 열고 어떤 코드를 작성해야 하는지 즉시 착수할 수 있어야 한다.

---

## 작성 규칙

### 포함해야 하는 것

- 참조 Plan (어떤 설계를 구현하는가)
- Phase 목록 (의존성 순서로 구성)
- 각 Phase 내 작업 상세:
  - 파일 경로 (신규/수정 여부)
  - 클래스·함수 시그니처
  - 주요 로직 (의사 코드 또는 핵심 알고리즘)
  - 테스트 케이스 (무엇을 검증하는가)
- Phase별 완료 조건
- 전체 완료 체크리스트

### 포함하지 않는 것

- 비즈니스 배경이나 "왜" 설명 (prd.md, plan.md 참조)
- 아키텍처 수준의 결정 (plan.md에서 이미 확정)
- 완성된 소스 코드 전체 (핵심 시그니처와 로직만)

---

## Phase 구성 원칙

Phase는 **구현 의존성 순서**로 나눈다.  
앞선 Phase가 완료되어야 다음 Phase를 시작할 수 있으며, 각 Phase가 끝나면 그 시점까지의 코드가 독립적으로 동작해야 한다.

Phase의 목적은 **한 번에 모든 레이어를 완성하지 않고**, 안쪽 레이어부터 검증하며 바깥쪽으로 확장하는 것이다.

| Phase | 대상 레이어 | 기준 |
|-------|------------|------|
| Phase 1 | 도메인 | 엔티티, Value Object, 도메인 서비스 |
| Phase 2 | 인프라 | Repository, External Client, DB 매핑 |
| Phase 3 | 애플리케이션 | UseCase / ApplicationService |
| Phase 4 | 프레젠테이션 | Controller, DTO, API 노출 |
| Phase 5 | 프론트엔드 | API 훅, 컴포넌트, 페이지 |

> 기능 성격에 따라 Phase를 합치거나 쪼갤 수 있다.  
> 예: 단순 CRUD라면 Phase 2+3을 합칠 수 있다.  
> 단, Phase를 건너뛰거나 순서를 역전시키지 않는다.

## 작성 팁

- 전체를 한 번에 구현하지 않는다. Phase 완료 조건을 충족한 뒤에만 다음 Phase로 넘어간다.
- 함수 시그니처는 실제 코드와 최대한 유사하게 작성한다. 구현 중 변경이 생기면 문서도 함께 업데이트한다.
- 이 프로젝트는 Kotest `DescribeSpec` 스타일을 사용한다. 테스트는 Spring 컨텍스트 없이 순수 단위 테스트로 작성한다.
- 완료된 Task는 `[x]`로 체크하고, Phase 상태는 작업 진행에 따라 즉시 업데이트한다.
