# PRD (Product Requirements Document) 가이드

## 목적

PRD는 **"무엇을 만드는가"** 에만 집중하는 문서다.  
구현 방법, 기술 스택, 코드 구조는 이 문서에 포함하지 않는다.

PRD가 완성되면 개발자와 기획자 모두가 "우리가 무엇을 만드는지" 동일하게 이해할 수 있어야 한다.

---

## 작성 규칙

### 포함해야 하는 것

- 배경과 문제 정의 (왜 이 기능이 필요한가)
- 목표 (이 기능으로 무엇을 달성하는가)
- 사용자 스토리 / 사용 시나리오
- 기능 요구사항 (사용자가 할 수 있어야 하는 것)
- 비기능 요구사항 (성능, 보안, 접근성 등)
- 범위 외 항목 (명시적으로 제외할 것)
- 성공 지표

### 포함하지 않는 것

- API 엔드포인트, HTTP 메서드, URL 경로
- 데이터베이스 스키마, 테이블명, 컬럼명
- 클래스명, 함수명, 패키지 구조
- 기술 스택 선택 이유
- 구현 순서나 스프린트 계획

## 템플릿

```markdown
# PRD: {기능명}

## 상태
- 작성일: YYYY-MM-DD
- 최종 수정: YYYY-MM-DD
- 상태: Draft | Review | Confirmed

---

## 배경

{이 기능이 필요하게 된 맥락과 문제 상황을 서술한다.}

## 목표

{이 기능을 통해 달성하려는 것을 명확하게 서술한다.}

## 사용자 스토리

- **{사용자 유형}**으로서, {목적}을 위해 {행동}하고 싶다.
- ...

## 기능 요구사항

### 필수 (Must Have)

- [ ] {요구사항 1}
- [ ] {요구사항 2}

### 선택 (Nice to Have)

- [ ] {요구사항}

## 비기능 요구사항

- **성능**: {예: 목록 조회는 1초 이내 응답}
- **보안**: {예: 인증된 사용자만 접근 가능}

## 범위 외

- {이번 구현에서 명시적으로 제외할 항목}

## 성공 지표

- {이 기능이 성공적으로 작동했다고 판단하는 기준}
```

---

## 작성 팁

- 기능 요구사항은 **사용자 행동** 단위로 작성한다. ("삼성전자를 검색할 수 있다" O, "검색 API를 만든다" X)
- 모호한 표현을 피한다. ("빠르게" → "1초 이내")
- 확인이 필요한 부분은 `[?]` 태그로 표시한다.
- PRD는 "왜"와 "무엇"에만 답한다. "어떻게"는 plan.md의 몫이다.
