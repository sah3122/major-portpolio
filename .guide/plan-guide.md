# Plan (Implementation Plan) 가이드

## 목적

Plan은 **"어떻게 구조화하는가"** 를 정의하는 문서다.  
PRD의 요구사항을 기술적으로 어떻게 구현할지 설계한다.  
코드 레벨 상세(함수 시그니처, 변수명 등)는 다루지 않는다.

Plan이 완성되면 개발자는 어떤 컴포넌트를 만들고, 어떤 흐름으로 데이터가 움직이는지 파악할 수 있어야 한다.

---

## 작성 규칙

### 포함해야 하는 것

- PRD 참조 (어떤 요구사항을 구현하는가)
- 아키텍처 결정 (레이어 구조, 모듈 분리 방식)
- 데이터 모델 (엔티티, 주요 필드, 관계)
- API 설계 (엔드포인트, 요청/응답 형태)
- 컴포넌트 구조 (프론트엔드의 경우)
- 의존성 관계 (어떤 모듈이 어떤 모듈을 사용하는가)
- 주요 설계 결정과 그 이유 (Trade-off)
- 제외한 대안과 이유

### 포함하지 않는 것

- 실제 코드 (의사 코드도 최소화)
- 함수 내부 로직
- 변수명, 상세 구현 절차
- 세부 Task 목록 (task.md의 역할)

## 작성 팁

- 설계 결정에는 반드시 "왜"를 함께 기록한다. 이유 없는 결정은 나중에 번복하기 어렵다.
- 불확실한 부분은 `[TBD]`로 표시하고 결정되면 업데이트한다.
- 도메인 모델과 DB 스키마를 구분해서 서술한다. (JPA 엔티티 ≠ 도메인 객체)
- Plan은 task.md 작성 전에 팀원과 리뷰해야 한다.
